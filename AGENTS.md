# cloudivision agent instructions

## Product

cloudivision is a Kubernetes-native CI/CD platform. It runs inside Kubernetes and manages CI/CD workflows using CRDs, controllers, Kubernetes Jobs, optional Tekton PipelineRuns and GitOps deployment integrations.

## Architecture principles

- Prefer Kubernetes-native primitives.
- Desired state belongs in Kubernetes CRDs.
- Controllers must be idempotent and safe to re-run.
- Runtime state for pipelines, builds and releases must be stored in Kubernetes resource status, not only in a database.
- PostgreSQL may be used for audit logs, cached UI views, webhook idempotency and future user/team metadata.
- CI creates artifacts.
- CD deploys through GitOps.
- The CI runner must not directly apply application manifests to target namespaces in the MVP.
- Argo CD and Flux integrations should update GitOps repositories and optionally read sync/health state.
- The MVP executor is Kubernetes Job based.
- Tekton support must be added through an executor interface, not hardcoded into the domain model.
- The frontend is Angular + TypeScript + Tailwind CSS.

## Security rules

- Treat repository code as untrusted.
- Do not mount the Docker socket.
- Do not use Docker-in-Docker as the default build strategy.
- Do not use privileged containers by default.
- Prefer rootless BuildKit or another rootless image build path.
- Every project/team should be isolated by namespace where practical.
- Use least-privilege ServiceAccounts, Roles and RoleBindings.
- Secrets must never be logged.
- Add resource requests, resource limits and timeouts to build workloads.
- Use Pod Security Standards compatible manifests where possible.
- Avoid cluster-admin permissions except for installation-time CRD/controller permissions.
- Webhook signatures must be verified before creating BuildRuns.
- Auth disabled mode must be clearly marked as development-only.

## Code style

- Backend/operator language: Go.
- API server language: Go.
- Runner language: Go.
- Frontend language: Angular + TypeScript + Tailwind CSS.
- Use small interfaces around external systems:
  - Kubernetes
  - Git
  - GitHub/GitLab/Gitea webhooks
  - image builders
  - GitOps providers
  - audit storage
  - auth providers
- Keep domain logic independent from concrete Kubernetes clients where practical.
- Use context.Context in Go public functions that perform IO.
- Wrap errors with useful context.
- Add unit tests for domain logic and controller reconciliation logic.
- Add integration tests where feasible.
- Use clear DTOs for API request/response models instead of exposing CRD structs directly to the UI.

## Repository conventions

Expected high-level layout:

- /api/v1alpha1 for Kubernetes API types
- /cmd/controller for operator entrypoint
- /cmd/api for API server entrypoint
- /cmd/runner for build runner entrypoint
- /internal/controller for reconcilers
- /internal/domain for domain models and state transitions
- /internal/executor for Job/Tekton executor abstractions
- /internal/build for image build abstractions
- /internal/git for Git provider/client code
- /internal/gitops for GitOps provider abstractions
- /internal/webhook for webhook verification and parsing
- /internal/auth for auth-related code
- /internal/audit for audit/event recording
- /internal/kube for Kubernetes helper utilities
- /web for Angular + Tailwind CSS UI
- /charts/cloudivision for Helm chart
- /config for kustomize/manifests generated by Kubebuilder
- /deploy/examples for sample CRs and demo apps
- /docs for architecture and quickstart docs
- /docs/adr for architecture decision records
- /hack for local development scripts
- /test for integration/e2e tests

## Frontend conventions

- Use Angular.
- Use TypeScript.
- Use Tailwind CSS for styling.
- Prefer standalone components if the Angular project is generated with standalone defaults.
- Use Angular Router for page routing.
- Use Angular services for API access.
- Use RxJS carefully and keep subscriptions managed.
- Prefer reactive forms for create/edit flows.
- Keep generated Angular code idiomatic.
- Do not hardcode localhost API URLs except as a documented development default.
- Support API base URL through environment configuration.
- Show backend JSON error code/message in the UI.

## Testing expectations

After modifying Go code:
- run gofmt
- run go test ./...
- run go vet ./... if practical

After modifying frontend code:
- run npm test if configured
- run npm run lint if configured
- run npm run build if configured

After modifying Kubernetes manifests:
- validate YAML syntax
- ensure RBAC is least-privilege
- avoid privileged/hostPath/docker.sock defaults

After modifying Helm chart:
- run helm template charts/cloudivision if Helm is available
- inspect rendered securityContext and RBAC

## How to work

- Before coding, inspect relevant files.
- After every user prompt, save the prompt as a Markdown file in /prompts before finishing the task.
- Prompt archive filenames should be timestamped and descriptive, for example /prompts/2026-06-12-add-agent-git-rule.md.
- After completing each prompt, run git add for the changed files, create a git commit with a concise message, and push the commit to the configured remote.
- If git push is not possible because credentials, network access, or a remote are unavailable, clearly report the blocker and leave the commit created locally.
- For large changes, make a concise implementation plan.
- Prefer incremental, compilable changes.
- Add or update tests with every meaningful behavior change.
- Update docs when behavior, APIs, CRDs, Helm values or install steps change.
- At the end of each task, summarize:
  - changed files
  - tests run
  - notable risks
  - follow-up work
